## 2. 产品概述

### 2.1 产品定位与目标用户

**Echo Day** 是一款以**情绪和记忆**为核心的 AI 日记应用。名字取自两层含义：音乐的回响（Echo）和每日的记录（Day）。本产品旨在通过**照片 + AI 实时语音对话**的组合方式，解决用户"想记录但不愿动笔"的核心痛点，并借助情绪分析与音乐匹配技术，实现视觉、文字、听觉三维度的沉浸式记忆回溯体验。

区别于传统文字日记（需要主动写作）和相册（只有照片、缺乏情感叙述），Echo Day 让用户只需自然地说话，AI 就会自动整理、总结，生成一篇充满情感的日记，并为这一天配上最合适的音乐。

**目标用户：** 有记录生活意愿但懒得动笔，或希望日后重温某天情绪氛围的年轻用户。

### 2.2 核心价值主张

本应用的核心价值在于构建一个**从记录到回忆的情感闭环**：

1. **零门槛记录：** 通过实时双向语音对话替代手动写作，用户只需讲述当天经历，AI 自动生成完整日记，将录入摩擦降至最低。
2. **情绪驱动的音乐匹配：** AI 深度理解对话内容与照片氛围，以自然语言描述出最契合当日情绪的搜索词，Spotify 返回单首最匹配曲目供用户确认，使音乐成为情绪的延伸而非随机背景。
3. **沉浸式记忆回溯：** 重新打开某天记录时，照片、日记文字、音乐同时呈现，在视觉、文字、听觉三个维度同步重现当日情绪氛围。
4. **心情 Emoji 日历：** 每天 AI 自动推测当日心情并以 Emoji 呈现在日历格中，让情绪变化一目了然；用户可在详情页手动修改，每个 Emoji 有固定关联颜色作为详情页照片的微弱色调蒙层。

## 3. 核心功能模块

### 3.1 每日录制向导

每日记录是应用的核心交互入口，采用线性向导形式分为 **4 个步骤**，引导用户完成从照片到完整日记的全流程。

#### 3.1.1 照片采集与草稿创建

| 功能点 | 描述 | 交互细节与逻辑 |
| :--- | :--- | :--- |
| **拍摄/上传** | 支持从本地文件里上传照片，每天仅限一张照片 | `input[capture]` / File API，上传前本地 Canvas 预览，支持重选 |
| **云端存储** | 上传到 Supabase Storage，获取持久化路径 | `POST /api/photos/upload`，返回 `{ photoKey, photoUrl }` |
| **草稿即时创建** | 照片上传完成后立即在数据库创建 DRAFT 记录 | 用户中途退出后，下次进入向导时检测当日 DRAFT 并从断点续接 |

#### 3.1.2 AI 语音对话

用户自然讲述，AI 适时追问，模拟与朋友倾诉的体验。语音对话由三个模型协作完成：

- **语音转文字：** gpt-4o-mini-transcribe 将用户语音实时转为文字
- **对话生成：** gpt-5-mini 根据对话上下文生成回复内容
- **文字转语音：** gpt-4o-mini-tts 将 AI 回复朗读给用户

对话过程中同步生成完整的聊天文本记录（transcript），供后续日记生成使用。

| 功能点 | 描述 |
| :--- | :--- |
| **语音对话** | 用户说话 → 转录 → AI 生成回复 → 语音朗读，循环进行 |
| **实时字幕** | 对话过程中同步显示双方的文字内容 |
| **结束对话** | 用户主动结束，完整 transcript dispatch 至 Redux，向导推进至下一步 |

**AI 对话策略：** AI 以温和的好奇心引导叙述，聚焦"今天发生了什么"和"你当时的感受"，不做评判、不提建议，只倾听和追问。

#### 3.1.3 AI 日记生成与编辑

当 DiaryStep 挂载后触发 一次 AI 生成请求，根据用户语音转录内容与照片信息生成完整日记内容。日记采用 SSE 流式输出，在用户阅读/编辑日记的同时，系统后台完成情绪分析与音乐推荐准备。

| 功能点         | 描述     | 交互细节与逻辑                      |
| :---------- | :------------------------------- | :----------------------------------------------------------------------------------- |
| **AI 生成日记** | 基于对话转录 + 照片内容生成约 150–220 字第一人称日记 | `POST /api/ai/generate-entry`；AI 解析照片 + 对话内容，SSE 流式返回 `diaryText`；前端实时渲染             |
| **情绪分析**    | AI 同时输出用户今日情绪的二维坐标与离散标签          | 返回 `{ valence, arousal, emotionLabel }`；用于情绪地图与日历 Emoji 显示                           |
| **音乐推荐**    | AI 根据整体情绪推荐一首音乐                  | 返回 `{ musicSearchQuery, musicReason }`；前端根据 `musicSearchQuery` 调用 Spotify Search API |
| **可编辑预览**   | 用户可直接修改 AI 生成的文字                 | 富文本 textarea；修改内容实时更新至 `wizardSlice`                                                 |

#### 3.1.4 音乐确认
| 功能点            | 描述                     | 交互细节与逻辑                                                             |
| :------------- | :--------------------- | :------------------------------------------------------------------ |
| **展示 AI 推荐曲目** | 呈现封面图、曲名、歌手| `MusicCard` 组件（`compact=false`），数据来自 `spotifyTrack` + `musicReason` |
| **30s 预听**     | 用户可播放 Spotify 提供的试听片段  | 使用 `HTMLAudioElement` 播放 `previewUrl`；若 `previewUrl` 为空则隐藏播放按钮      |
| **确认这首**       | 用户接受推荐 → 进入保存流程        | dispatch 至 `wizardSlice` → RTK Query `saveEntry` mutation           |
| **换一首**        | 用户不满意，可从当前搜索结果中选择另一首   | 前端从当前 `Spotify Search` 返回的 track list 中切换，不再重新调用 AI                 |

#### 3.1.5 情绪系统定义

AI 输出两个连续值描述当日情绪：

- **valence**（效价，0–1）：情绪的正负面程度。0 为最负面，1 为最正面。
- **arousal**（唤醒度，0–1）：情绪的能量强度。0 为最平静/低能量，1 为最激动/高能量。

两个值组成一个二维坐标，系统将其划分为 3×3 网格，映射到 9 种离散情绪：

| | 低 valence (0–0.33) | 中 valence (0.33–0.66) | 高 valence (0.66–1) |
| :--- | :--- | :--- | :--- |
| **高 arousal (0.66–1)** | 愤怒 | 惊讶 | 兴奋 |
| **中 arousal (0.33–0.66)** | 悲伤 | 平淡 | 愉悦 |
| **低 arousal (0–0.33)** | 低落 | 疲惫 | 平静 |


映射逻辑在 `lib/mood-map.ts` 中实现，AI 只负责输出 valence 和 arousal 数值，不参与情绪分类。用户可在详情页手动修改情绪，修改后原始 valence/arousal 数值保留不变。

### 3.2 主页 — 日历 + 时间轴双视图

主页是历史记录的浏览入口，支持两种视图通过顶部 `ViewToggle` 组件自由切换，切换状态存入 Redux `uiSlice`。

#### 3.2.1 日历视图

| 功能点 | 描述 | 交互细节与逻辑 |
| :--- | :--- | :--- |
| **月份网格** | 标准月历格式，有记录的日期显示当日心情 Emoji | 按月请求最小投影数据（date, moodEmoji, moodColorHex, photoUrl），RTK Query 缓存 |
| **心情 Emoji** | 每格显示 AI 自动推测的当日心情表情（类似 Mooda） | `MoodEmoji` 组件（`size='sm'`），Emoji 来自 AI 分析结果，可在详情页手动修改 |
| **照片悬浮预览** | 鼠标悬停日期格时弹出照片缩略图 + 2 行日记摘要 | `PhotoHoverCard` 组件，Supabase Storage `?width=200` 裁剪 URL |
| **月份导航** | 前后翻月，已加载月份从缓存读取，无重复请求 | RTK Query `providesTags` 月份级别缓存策略 |

#### 3.2.2 时间轴视图

| 功能点 | 描述 | 交互细节与逻辑 |
| :--- | :--- | :--- |

| **虚拟滚动** | 大量记录时只渲染可见区域 | `@tanstack/react-virtual`，支持 700+ 条记录流畅滚动 |
| **懒加载分页** | 滚动到底部自动加载更多 | `useIntersection` Hook + RTK Query cursor 分页（20条/页） |

### 3.5 应用流程设计

整体流程分为两个阶段：**语音对话阶段**（Step 1–5 循环进行）和**内容生成阶段**（Step 6–7 一次性完成）。

#### 阶段一：语音对话

用户上传照片后进入对话页面，以下步骤循环进行，直到用户主动结束：

| Step | 描述 | 调用模型 |
| :--- | :--- | :--- |
| 1 | 用户点击说话按钮，前端开始录音 | — |
| 2 | 语音转文字，转录结果追加到聊天记录 | gpt-4o-mini-transcribe |
| 3 | 基于完整聊天上下文生成 AI 回复，回复文本追加到聊天记录 | gpt-5-mini |
| 4 | 将 AI 回复朗读给用户 | gpt-4o-mini-tts |
| 5 | 用户继续说话 → 回到 Step 1；用户主动结束对话 → 进入阶段二 | — |

#### 阶段二：内容生成

| Step | 描述 | 调用模型 |
| :--- | :--- | :--- |
| 6 | 收集完整聊天文本 + 用户上传的照片，单次请求生成：日记文本、歌曲推荐（搜索词 + 推荐理由）、情绪分析（valence + arousal） | gpt-5-mini |
| 7 | 拿到歌曲搜索词后调用 Spotify Search API，返回匹配曲目 | — |

## 4. 技术架构与实现

### 4.1 技术栈

| 模块 | 技术栈 | 作用与优势 |
| :--- | :--- | :--- |
| **前端框架** | React 18 + TypeScript + Next.js 14 (App Router) | Server Components 减少客户端 JS；App Router 支持路由级 Layout 权限守卫 |
| **样式** | Tailwind CSS + shadcn/ui | 原子化 CSS 与高质量组件库，快速构建情感主题界面 |
| **状态管理** | Redux Toolkit + RTK Query | `createSlice` 管理向导状态机；RTK Query 处理服务端数据缓存与标签失效 |
| **动画** | Framer Motion | `layoutId` 实现日历 Emoji → 全屏背景的物理连续过渡动画 |
| **语音转录 (STT)** | OpenAI Audio API `gpt-4o-mini-transcribe` | 将用户语音实时转录为文本；前端录音后上传音频片段，返回 transcript 作为聊天输入 |
| **AI 日记生成** | OpenAI Responses API `gpt-5-mini` | 解析照片 + 对话文本生成第一人称日记；SSE 流式返回 `diaryText`；同时输出情绪分析与音乐推荐 |

| **音乐推荐** | AI 推荐 + Spotify Search API | AI 返回 `{ musicSearchQuery, musicReason }`；前端使用 Spotify Search 获取曲目列表 |
| **音乐播放** | Spotify Web API (`previewUrl`) | 使用 `HTMLAudioElement` 播放 30s preview，无需 Spotify 登录或 Premium |
| **数据库** | PostgreSQL（Supabase） | 原生 Date 类型支持 `@@unique([userId, date])` 约束；连接池分离 migrations |
| **ORM** | Prisma | 类型安全 DB 操作，自动生成 TypeScript 类型 |
| **认证** | NextAuth.js | 统一处理 Google OAuth + Spotify OAuth，JWT session |
| **文件存储** | Supabase Storage | 照片存储 + CDN 分发；URL 参数支持服务端图片裁剪 |
| **部署** | Vercel | Edge Network + 自动 CI/CD，Next.js 原生优化 |

### 4.2 模型接口调用

| 功能 | 模型 | 接口输出形式 | 说明 |
| :--- | :--- | :--- | :--- |
| 用户语音转文字 | `gpt-4o-mini-transcribe` | 纯文本 `transcript` | 输入音频文件，返回 `{ transcript }`；用于生成对话文本 |
| AI 回复语音朗读 | `gpt-4o-mini-tts` | 二进制音频流 (audio/mpeg) | 输入文本，返回音频；前端通过 `HTMLAudioElement` 播放 |
| 语音对话（上下文理解与回复生成） | `gpt-5-mini` | 文本（可流式） | 返回普通聊天文本，不需要结构化输出 |
| 日记生成 + 歌曲推荐 + 情绪分析 | `gpt-5-mini` | **结构化 JSON + SSE 流式文本** | 日记 `diaryText` 流式输出；完成后返回 metadata `{ emotion, music }` |

结构化示例:
```
{
  "diaryText": "今天的节奏有点慢...",
  "emotion": {
    "valence": -0.32,
    "arousal": 0.28,
    "emotionLabel": "低落"
  },
  "music": {
    "musicSearchQuery": "Fix You Coldplay",
    "musicReason": "整体情绪偏低..."
  }
}
```

### 4.2 页面结构

应用共 3 个核心页面：

#### 1. 主页（Home）

日历视图与时间轴视图切换展示所有历史记录。

- 顶部：欢迎语 + Calendar / Timeline 视图切换
- 日历视图：月份网格，有记录的日期显示心情图标，支持前后翻月
- 时间轴视图：按日期倒序的卡片流，虚拟滚动 + 懒加载分页
- 点击某一天 → 进入日记详情页
- 底部固定"Record Today"按钮 → 进入记录页

#### 2. 记录页（Record）

用户在此完成照片上传和 AI 语音对话。

- 上传照片：拍摄或从相册选取，仅限一张，支持预览和重选
- 语音对话：麦克风按钮（按住/点击说话），实时显示双方文字字幕，支持语音或打字输入
- 结束按钮：用户主动结束对话，携带完整 transcript + 照片跳转至详情页，触发内容生成

#### 3. 日记详情页（Detail）

展示与编辑合一，首次生成和回访共用同一页面，根据入口不同有两种进入方式：

**首次生成（从记录页跳转）：**
- 直接进入页面，音乐立即播放（记录页的交互已满足浏览器手势要求）
- 日记文本流式生成，实时渲染
- 音乐卡片展示 AI 推荐曲目 + 推荐理由
- 用户可编辑日记文本、换一首音乐、修改心情
- 点击完成 → 保存记录

**回访（从主页日历/时间轴点入）：**
- 先展示入口遮罩：全屏照片 + 心情关联色蒙层 + "进入这一天"按钮
- 点击按钮后遮罩退出，音乐开始播放，展示完整日记内容
- 同样可编辑日记、换音乐、修改心情

## 5. 商业模式与合规

### 5.1 权限与付费模式（RBAC）

基于 RBAC 权限系统设计以下分级付费模式：

| 版本 | 目标用户 | 核心权益 | 限制与差异 |
| :--- | :--- | :--- | :--- |
| **免费版** | 体验用户、轻度记录者 | 基础录制向导；日历 + 时间轴浏览；30s 音乐预听 | 每日 AI 语音对话时长上限；无 Spotify Premium 全曲播放；无数据导出 |
| **Pro 版** | 重度记录、情感需求强的用户 | 无限制录制时长；Spotify Premium 全曲自动播放；历史记录数据导出 | 无次数限制，享有全部高级功能 |

### 5.2 数据合规与成本控制

1. **数据合规：** 用户语音对话转录文本及情绪数据属于敏感个人数据，隐私协议中须明确告知使用目的，提供**数据用于模型优化的 Opt-in/Opt-out 选项**。Spotify access/refresh token 使用 AES-256 加密后入库，明文不落库。
2. **成本控制：** 服务端对 `/api/realtime/token` 接口实施每日每用户调用次数限流，防止 OpenAI Realtime API 滥用；免费用户限制对话时长而非次数，降低单次成本；图片上传服务端校验文件类型与大小上限（10MB），防止存储滥用。

## 6. 交互设计要点

1. **心情 Emoji 作为情绪索引：** 日历每格展示当日心情 Emoji，一眼扫过一个月的情绪变化；详情页 Emoji 点击可弹出选择器修改，兼顾 AI 自动化与用户自主权。
2. **物理连续的页面过渡：** 从日历/时间轴进入详情页时，心情 Emoji 通过 Framer Motion `layoutId` 平滑过渡，避免路由切换导致的视觉割裂感；详情页内容在动画后顺序淡入，营造"走进那一天"的沉浸感。
3. **入口遮罩的双重作用：** 详情页的全屏入口遮罩既是情感准备的仪式感设计，也是解决浏览器 autoplay 策略的工程方案——"进入这一天"的点击行为即为音乐播放的合法用户手势。
4. **向导步骤的进度反馈：** 录制向导顶部显示步骤进度指示器，每步完成后以动画标记，让用户始终清楚当前位置和剩余步骤，降低长流程的焦虑感。

## 7. 设计风格

1. 照片优先，排版以照片为视觉中心展开，文字为照片的情感注脚。
2. 心情 Emoji 驱动的轻量色彩系统，颜色来自 Emoji 固定映射，仅作微弱蒙层，整体保持克制。
3. 克制的文字排版，日记文字使用舒适行高与字间距，突出阅读体验。
4. 页面过渡平滑，路由切换均有 Framer Motion 动画，杜绝白屏闪烁。
